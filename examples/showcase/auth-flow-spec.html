Authentication Flow v2

Magic-link login with optional WebAuthn step-up. Replaces the existing OAuth-only flow that legal flagged in Q2.

Owner · Maail Hasan Reviewers · Eng, Security, Legal Target · 2026-06-30 Draft

Flow

Why this changes

The existing OAuth-only flow ties identity to a third-party IdP. Legal flagged token retention in Q2 — we cache provider tokens in a way that doesn't meet the new compliance requirements. The migration is compliance-driven, not tech-debt cleanup.

Decisions

  • ✓ Email magic links via Loops, 15-minute TTL
  • ✓ WebAuthn (passkey) for step-up only — billing changes, team-admin actions, account deletion
  • ✓ Drop Google/GitHub OAuth — keep them as identity hints only
  • ⚠ Session storage: server-side only, 30-day rolling, http-only cookies

Endpoints

MethodPathPurpose
POST/auth/requestSend magic link to email
GET/auth/verify?token=…Exchange token for session
POST/auth/webauthn/beginStart passkey ceremony
POST/auth/webauthn/finishVerify and elevate session
POST/auth/logoutRevoke active session

Open questions

  • Do we honor Sign in with Apple's email-relay tokens for legacy users?
  • Is 15-minute TTL too aggressive for users on enterprise email security gateways?
  • Recovery flow when a user loses their device + their email — out of scope?
